# Linting Rules and Quality Checklist

This document defines the linting rules and quality standards for this project. Contributors (both human and AI) should follow these guidelines for all code and documentation contributions.

## Documentation Standards

### Markdown Files

- All Markdown files must pass linting checks before being committed
- Use the `text` language specifier for conversation code blocks 
- Use appropriate language specifiers for all code blocks (e.g., `bash`, `javascript`, `json`)
- Ensure proper spacing around code blocks (blank lines before and after)
- Ensure lists are surrounded by blank lines
- Use consistent heading levels (increments of one level at a time)

### Code Standards

- JavaScript code should follow ESLint rules
- Follow consistent indentation (2 spaces recommended)
- Include JSDoc comments for functions
- Use consistent naming conventions

## Checklist for All Contributions

Before completing any task, verify:

1. No linting errors remain in any modified files
2. Code changes include appropriate tests
3. Documentation is updated to reflect code changes
4. No duplicate content exists in documentation

## For AI Assistants

AI assistants must:

1. **Always check for warnings/errors after making changes** using the `get_errors` tool
2. Fix all linting issues before considering a task complete
3. Report to the user if any warnings cannot be resolved and explain why
4. Never leave duplicate content in documentation
5. Ensure all code blocks have appropriate language specifiers

## How to Check for Errors

```bash
# For AI assistants (using tools)
get_errors(["/path/to/file1", "/path/to/file2"])

# For human contributors
npm run lint
```

This document serves as a reminder and checklist to maintain high-quality standards across the project. All contributions should follow these guidelines to ensure consistency and readability.
